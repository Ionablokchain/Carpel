# Carpel Compiler Architecture

This document describes the internal design of the Carpel compiler (code name `carpelc`). It is intended for contributors who want to understand how the compiler works, where to find specific functionality, and what the main design trade-offs are.

## High-Level Overview

`carpelc` translates Carpel source code into native machine code via LLVM. It performs full region inference to guarantee memory safety without lifetime annotations, and it incorporates the runtime directly into the compiled binaries (no external runtime dependency beyond a small static library).

The compiler is written in **Rust** (for now) and leverages:
- [`inkwell`](https://github.com/TheDan64/inkwell) – safe Rust bindings for LLVM.
- A hand-written recursive descent parser (no parser generator for maximum control over error messages).
- A custom region solver based on the algorithm sketched in the language manual.

## Source Representations

### 3.1 Token Stream
The lexer converts source text into a stream of tokens. Each token stores its kind, text, and source location. Keywords (`fn`, `let`, `arena`, `ref`, `mut`, etc.) are recognised, as are identifiers, literals, operators, and punctuation. Whitespace and comments are discarded.

Key file: `src/lexer.rs`

### 3.2 Abstract Syntax Tree (AST)
The parser produces a concrete syntax tree owned by the AST arena. Nodes are defined as Rust enums in `src/ast.rs`. Important AST nodes include:
- `Item::Fn(FnDef)`, `Item::Struct(StructDef)`, etc.
- `Stmt::Let(LetStmt)`, `Stmt::Arena(ArenaBlock)`, …
- `Expr::Literal(Literal)`, `Expr::Path(Path)`, `Expr::Call(Box<Call>)`, …

The AST is not yet type‑checked and retains all syntactic details (including parentheses) for error reporting.

### 3.3 High-Level IR (HIR)
After name resolution and region inference, the AST is lowered to HIR. HIR removes syntactic sugar, desugars `if`/`loop` into simpler control flow, and explicitly represents regions.

Key structures:
- `HirId` – unique identifier for every item, expression, and pattern.
- `HirCtx` – a context containing all HIR nodes, the region graph, and type information (after type checking).
- Region variables are assigned to every expression that produces a reference; the region inference pass updates these.

### 3.4 Mid-Level IR (MIR)
MIR is a control‑flow graph representation used for borrow checking, optimisation, and eventual LLVM IR generation. It makes memory management explicit: `alloca` for stack slots, `arena_alloc` intrinsics for arena allocations, and explicit drops.

The MIR is a simple three‑address code variant with basic blocks, terminators, and place/value operands. It is designed to be easily lowered to LLVM IR.

## Key Compiler Passes

### 4.1 Lexing and Parsing
- **Lexer** (hand‑written): takes a `&str` and returns an iterator of tokens.
- **Parser** (recursive descent): uses a single‑token lookahead and constructs an AST. Error recovery is minimal; on failure it attempts to synchronise to the next statement boundary.

Relevant crates/modules:
- `carpelc_lexer`
- `carpelc_parser`

### 4.2 Name Resolution
Resolves paths (`std::collections::Vec`, local variables, etc.). Builds a map from identifiers to definition IDs. Imports (`use`) are expanded. The resolver also detects name conflicts and undeclared identifiers.

### 4.3 Region Inference
This is the most Carpel‑specific pass. It infers region variables and constraints without any user‑provided lifetime annotations.

**How it works:**
1. Assign a fresh region variable to every reference‑producing expression and to every allocation site (stack slot, arena alloc).
2. Walk the HIR generating constraints:
   - For assignment `x = &y` (or `x = y` where `y` is a reference), constraint `region(x) ⊇ region(y)` (the target must outlive the source).
   - For function calls, the actual argument regions must be subsets of the formal parameter regions (contravariant).
   - For returns, the returned region must outlive the function’s return region.
3. Solve the set of constraints using a fast union‑find with a topological ordering. If a cycle is detected that would require a region to live longer than another region that is definitely shorter, report an error.

This algorithm is a simplified version of the one used in Rust’s NLL. Because Carpel doesn’t have explicit lifetimes, the constraint graph is usually simpler and can be solved quickly.

### 4.4 Type Checking
Type checking follows region inference so that types (which may involve regions) are fully known. The type checker verifies:
- Expression types match expected types.
- Trait bounds are satisfied.
- Match exhaustiveness.
- The `Copy` trait is respected in move contexts.

After type checking, the HIR contains full type information, including concrete region assignments.

### 4.5 Lowering to MIR
This pass converts HIR to MIR:
- Destructures pattern matching into conditional branches.
- Converts `arena` blocks into `arena_begin`/`arena_end` intrinsics and internal `arena_alloc` operations.
- Makes control flow explicit with basic blocks.

The MIR is then used for a final borrow check to ensure no aliasing rules are violated. Because region inference already guarantees scoping, this check mainly validates the `ref`/`mut` exclusivity at the MIR level.

### 4.6 LLVM Code Generation
The MIR is translated into LLVM IR using `inkwell`. Key mappings:
- Stack‑allocated variables → `alloca`.
- Arena allocations → GEP on the arena buffer with an offset bump (see Runtime Library).
- `Shared<T>` operations → calls to the `Shared` runtime API.
- `async`/`await` → state machines with task‑local arenas (the compiler emits calls to the async runtime).

The resulting LLVM module is then passed to the LLVM optimisation pipeline and finally emitted as an object file.

## Runtime Library

The Carpel runtime (`libcarpl_rt`) is a static library linked into every binary. It is written partly in Rust and partly in C (for maximum portability). The runtime provides:

### 5.1 Arena Allocator
The `arena` language construct compiles to direct manipulation of an `Arena` struct defined in the runtime. The struct holds a pointer to a byte buffer, its capacity, and a current offset. The compiler emits inline GEP + store instructions for every `arena.alloc`, which is extremely fast and avoids function call overhead.

When an arena block ends, the compiler simply drops the `Arena` local; no destructor iteration is needed.

### 5.2 Shared State
`Shared<T>` is a smart pointer implemented as an `Arc<Mutex<T>>` in the runtime. The compiler emits calls to `Shared::new` and `Shared::lock` where appropriate. Because Carpel does not allow direct dereferencing of a `Shared<T>` without `lock()`, the compiler statically prevents data races.

### 5.3 Async Runtime
A lightweight, single‑threaded cooperative scheduler lives in `libcarpl_rt::task`. Each `Task<T>` is a heap‑allocated future, but the compiler may optimise small tasks to stack allocation. The runtime provides a `block_on` function for executing an async main, and low‑level primitives for timers and I/O (based on `epoll`/`kqueue`/`IOCP`).

## Design Decisions

- **No parser generator** – Hand‑written recursive descent gives us full control over diagnostic quality and prevents dependency bloat.
- **Separate HIR and MIR** – HIR retains syntactic structure for error messages; MIR is closer to LLVM IR and enables optimisations and straightforward code generation.
- **Arena allocation inlined** – Avoiding function calls for `arena.alloc` is critical for performance. The runtime provides only the buffer management; the compiler generates the access logic directly.
- **Region inference without user annotations** – This keeps the language surface extremely simple. The trade‑off is a slightly more complex solver, but it’s still polynomial and fast in practice.
- **Unsafe code isolated** – The compiler, parser, and code generation are all safe Rust except for the LLVM bindings and a few platform‑specific intrinsics. This limits the risk of bugs in the compiler itself.

## Future Work

- **Self‑hosting** – Rewrite the compiler in Carpel itself once the language is stable.
- **Incremental compilation** – Exploit the region and type information for fast re‑compilation.
- **LLVM alternative backends** – Possibly a Cranelift backend for faster debug builds.
- **Formal verification of the region solver** – Provide a proof that the inference algorithm is sound.
