1. The complete EBNF grammar of the Carpel language
This grammar covers the entire Carpel language (v0.1.0), including the unique concepts: arenas, ref/mut references, async/await, and unsafe. We used traditional EBNF notation, with:

[...] – optional

{...} – repetition (zero or more)

(...) – grouping

| – alternative

terminals written between straight quotes "..."

General structure
ebnf
program = { top-level-item } ;

top-level-item = fn-def | struct-def | enum-def | trait-def | impl-block
| const-def | use-decl | extern-block
;

use-decl = "use" path ";" ;
path = ident { "::" ident } ;
Definitions
ebnf
fn-def = "fn" ident [ generic-params ] "(" [ param-list ] ")" [ "->" type ] block ;

param-list = param { "," param } ;
param = identity ":" type ;

generic-params = "<" generic-param { "," generic-param } ">" ;
generic-param = ident [ ":" trait-bounds ] ;

trait-bounds = trait-ref { "+ trait-ref } ;
trait-ref = ident [ "<" type-list ">" ] ;

struct-def = "struct" ident [ generic-params ] struct-body ;
struct-body = "{" { field ";" } "}" | ";" ;
field = identity ":" type ;

enum-def = "enum" ident [ generic-params ] "{" variant { "," variant } [ "," ] "}" ;
variant = ident [ tuple-type ] | ident "{" { field ";" } "}" ;

trait-def = "trait" ident [ generic-params ] "{" { trait-item } "}" ;
trait-item = fn-sig | type-alias ;
fn-sig = "fn" ident [ generic-params ] "(" [ param-list ] ")" [ "->" type ] ;
type-alias = "type" ident ";" ;

impl-block = "impl" [ generic-params ] trait-ref ? "for" type "{" { fn-def } "}" ;

const-def = "const" ident ":" type "=" expression ";" ;

external-block = "extern" string-literal "{" { external-fn } "}" ;
external-fn = "fn" ident "(" [ param-list ] ")" [ "->" type ] ";" ;
Types and expressions
ebnf
type = base-type | "ref" base-type | "mut" base-type ;
base-type = ident [ "<" type-list ">" ] 
| "(" type ")" 
| tuple-type 
| fn-type 
;
type-list = type { "," type } ;
tuple-type = "(" type { "," type } ")" ;
fn-type = "fn" "(" [ type-list ] ")" [ "->" type ] ;

expression = literal 
| path 
| operator-expression 
| call-expression 
| field-access 
| tuple-expr 
| block-expr 
| if-expr 
| match-expr 
| loop-expr 
| while-expr 
| for-expr 
| return-expr 
| arena-block 
| async-block 
| unsafe-block 
| closure 
;

literal = integer-literal | float-literal | string-literal | bool-literal | unit-literal ;

block-expr = block ;
block = "{" { statement } "}" ;

statement = let-stmt | expression ";" | item

let-stmt = "let" [ "mut" ] ident [ ":" type ] "=" expression ";" ;

if-expr = "if" expression block [ "else" ( block | if-expr ) ] ;
match-expr = "match" expression "{" { match-arm } "}" ;
match-arm = pattern [ "if" expression ] "=>" ( expression "," | block ) ;

loop-expr = "loop" block ;
while-expr = "while" expression block ;
for-expr = "for" pattern "in" expression block ;

return-expr = "return" [ expression ] ";" ;

arena-block = "arena" ident ":" "Arena" "(" expression ")" block ;

async-block = "async" block ; // async closure or block (for Task)

unsafe-block = "unsafe" block ;

closure = "|" [ closure-param-list ] "|" [ "->" type ] block ;
closure-param-list = ident [ ":" type ] { "," ident [ ":" type ] } ;

call-expression = expression "(" [ expression-list ] ")" ;
expression-list = expression { "," expression } ;

field-access = expression "." (ident | integer-literal ) ;

operator-expression = expression binary-operator expression 
| unary-operator expression
;
binary-operator = "+" | "-" | "*" | "/" | "%" | "==" | "!=" | "<" | ">" | "<=" | ">="
| "&&" | "||" | "&" | "|" | "^" | "<<" | ">>" | "as"
;
unary-operator = "-" | "!" | "*" | "&" | "&mut" ;
Notes:

"ref" and "mut" are reserved keywords in type position; they cannot be used as normal identifiers.

"arena" is recognized as a special statement that creates a new scope.

Async functions are written as regular functions with the async fn prefix.

2. Interaction table of ref and mut references
This table summarizes which references can coexist at a given point in the program, respecting the fundamental rule: a mut T is an exclusive reference; no other access (read or write) is allowed while it exists.

Action / Context ref T active mute T active
Create ref T (shared borrowing) Allowed as many as (multiple reads) Forbidden – you cannot create a ref when a mute already exists
Create mute T (exclusive borrowing) Forbidden – if there is an active ref, you cannot create a mute Allowed only one (exclusivity)
Read via ref T Allowed Not applicable (does not exist)
Write via mute T Not applicable (does not exist) Allowed
Transfer of ownership (move) Forbidden as long as there are active (shared or exclusive) references Forbidden as long as the reference is active
Lifetime Can be shorter than the referenced data; cannot survive data Likewise, it cannot survive data and cannot coexist with other references
These rules are statically checked by the compiler's built-in borrow checker (after the region inference stage). There is no RefCell to bypass these rules; if you need shared mutability, use Shared<T> (intended for concurrency, but can also be used on a single thread, although it is more cumbersome).

Example of a common error:

carpel
let mut data = 42;
let r1: ref i32 = &data;
let r2: mut i32 = &mut data; // ERROR: cannot borrow mutable while
// there are active shared references
Method case:
When a method receives a ref self, you receive a ref Self (read access); when it receives mut self, you receive a mut Self (exclusive write access). You cannot call a method that requires mut self if you only have a ref reference.

3. The async rule system – “Implicit Pinning”
In Carpel, asynchronous programming never exposes the concept of Pin. The compiler automatically takes care of address stability for local variables that are referenced after a suspension point (.await). This approach is called “implicit pinning” and is based on a micro-arena associated with each task.

How it works
Transforming an async function into a state machine:
An async fn function is transformed by the compiler into an anonymous type that implements the Task internal trait. Each .await point becomes a state transition.

Detecting self-referential variables:
During flow analysis, the compiler identifies all local variables that:

Are declared with let mut (or are implicitly mutable) and

Are referenced (by ref or mut) at a point before an .await, and that reference is used after the .await.

These variables are called suspended-referenced variables.

Relocating variables in a per-task micro-arena:
For each such variable, the compiler:

Allocates the required memory within a micro-arena attached to the state machine (called the task-local arena).

Replaces the variable on the stack with an index/handle to that location in the arena.

Thus, the address of the variable remains fixed throughout the lifetime of the task, even if the state machine is moved in memory (for example, when placed in the executor queue).

Task-local micro-arena:
This arena is an internal buffer allocated on the heap (or stack for small tasks) at task initiation. It grows monotonically only as needed, and is freed automatically when the task ends. The model is similar to a bump allocator, but completely managed by the compiler – without programmer intervention.

Costs:

For variables that are never referenced over .await, there is no cost: they remain on the automaton stack.

For relocated ones, there is a single allocation inside the micro-arena and indirect accesses (via pointers). These are rare in practice, since most async code does not self-reference data across await points.

Safety:
The compiler guarantees through region checks that no reference created inside the task can escape outside it. Thus, the micro-arena and all its data are freed correctly upon task completion, without the possibility of dangling pointers.

Example
carpel
async fn example() -> i32 {
let mut x = 42;
let r = &mut x; // exclusive reference
temporary().await; // suspension – r is now at a dangerous point
*r += 1; // use after await
x
}
Behind, the compiler produces (pseudocode):

Before the first await: move x from the stack to task_arena, update r as a pointer to that location.

After the await: reload r from the state machine (which kept the pointer) and continue.
